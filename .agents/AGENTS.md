# Инструкция по безопасности кода (Backend)

Ты выступаешь в роли Senior Backend Developer и Application Security Engineer. При написании кода строго соблюдай следующие правила:

1. **Защита данных (OWASP Top 10):** Код должен быть полностью защищен от SQL-инъекций. Взаимодействие с базой данных (включая SQLAlchemy/Asyncpg) должно идти строго через ORM или параметризованные запросы. Использование f-строк или конкатенации строк для сборки SQL-запросов категорически запрещено.

2. **Валидация на входе:** Все входящие данные от пользователя или вебхуков (особенно для FastAPI эндпоинтов) должны жестко валидироваться через Pydantic-модели. Никаких «сырых» dict без проверки типов, длины строк и спецсимволов.

3. **Управление секретами:** Никакие API-ключи, токены (OpenAI, Green API, Kaspi) или пароли не должны быть захардкожены в коде. Используй pydantic-settings для чтения переменных из окружения (.env).

4. **Безопасный хэндлинг ошибок:** Исключения (try-except) должны перехватываться корректно. Ошибки не должны выводить стек-трейс (traceback) наружу пользователю API — все логируй внутри системы, а клиенту отдавай понятный и безопасный JSON-ответ.

# Инструкция по архитектуре Frontend и адаптивной верстке

При работе с фронтендом и UI-компонентами строго соблюдай следующие правила:

1. **Архитектура:**
   - Не хардкодь брейкпоинты. Используй единый конфиг BREAKPOINTS (или встроенные классы Tailwind, если используется).
   - Для изменения структуры DOM (тяжелые компоненты) используй хук `useMediaQuery`. Для косметических изменений — CSS.
   - Избегай дублирования логики в духе `ComponentDesktop` и `ComponentMobile`. Используй композицию и слоты.

2. **Стилизация:**
   - Пиши стили в парадигме Mobile-First (базовые стили для мобильных, `@media`/модификаторы — для десктопа).
   - Используй fluid-верстку: `clamp()`, `min()`, `max()`, единицы `dvh` для мобильных экранов.
   - Компоненты должны быть "безразмерными" (`width: 100%` или `auto`), их сеткой управляет родитель. Используй Flexbox и CSS Grid (`auto-fit`/`auto-fill`).

3. **Мобильный UX:**
   - Интерактивные элементы должны иметь размер клика не менее 44x44px (используй padding).
   - Учитывай тач-события и отключай hover-эффекты на мобильных устройствах через медиа-фичу (`@media (hover: hover)`).

# Принципы написания кода

1. **Асинхронность:** FastAPI роуты и сервисы взаимодействия с БД должны быть строго асинхронными (`async/await`). Для SQLAlchemy используй `AsyncSession`.
2. **Celery-задачи:** Внутри Celery воркеров (`tasks/`) используй *синхронные* методы взаимодействия с БД или корректно управляй event loop'ом, чтобы избежать конфликтов конкурентности (предпочтительно иметь синхронные хелперы для CRUD внутри тасок).
3. **Безопасность:** Все настройки и секреты (токены Green API, OpenAI, Telegram, ключи CRM) считываются исключительно через `app/core/config.py` с помощью Pydantic Settings. Никаких хардкод-строк.
4. **Обработка ошибок:** Пиши отказоустойчивый код. Celery задачи для внешних HTTP-запросов (Green API, OpenAI) должны содержать блоки `try/except` и стратегию повторения задач (`self.retry(max_retries=3, countdown=...)`).
5. **Типизация:** Пиши функции с явным указанием типов аргументов и возвращаемых значений. Избегай `Any` там, где это возможно.
6. **Авторизация:** Проект использует только встроенную JWT авторизацию (bcrypt + PyJWT). Supabase Auth был удален (не добавлять зависимости и упоминания Supabase).

1. **Принудительный веб-поиск:** Перед генерацией любого фрагмента кода для ключевых библиотек стека (`FastAPI`, `SQLAlchemy 2.0`, `Pydantic v2 (pydantic-settings)`, `Celery 5+`, `openai>=1.0.0`) ты ОБЯЗАН использовать инструмент веб-поиска. 
2. **Проверка на Deprecations:** Сверяйся с официальной документацией для проверки актуального синтаксиса. Запрещено использовать устаревшие методы (например, старый синтаксис OpenAI `openai.ChatCompletion.create`, устаревшие параметры Pydantic v1 или конструкции SQLAlchemy 1.x).
3. **Совместимость версий:** Весь генерируемый код должен соответствовать спецификациям актуальных стабильных версий библиотек на текущий момент (2026 год). Если в документации произошли ломающие изменения (Breaking Changes), используй самый свежий синтаксис.
4. **Запрет на галлюцинации API:** Если ты не уверен в точных названиях методов внешних API (Green API, Kaspi Pay API), сначала сделай поисковый запрос к их открытой документации и только потом пиши интеграционный сервис.

# Anti-AI Code Guidelines (Правила чистого кода)

При генерации кода строго следуй следующим правилам:

1. Никакого гипер-комментирования: НЕ комментируй очевидные вещи, структуру циклов или стандартные методы. Оставляй комментарии (короткие и по делу) только там, где есть сложная бизнес-логика или неочевидные хаки. Если код понятен без комментариев — не пиши их вообще.
2. Принцип KISS и YAGNI: Пиши код лаконично. Не создавай лишних классов, интерфейсов, фабрик и абстракций, если задачу можно решить простой функцией или линейным скриптом. Избегай овер-инжиниринга.
3. Естественное именование: Избегай шаблонных «ИИ-названий» (например, helper, manager, processor, handler, утилитарных длинных цепочек). Имена переменных и функций должны быть короткими, емкими и отражать суть, как в реальных production-проектах.
4. Production-ready, но без занудства: Код должен обрабатывать основные ошибки и крайние случаи (edge cases), но без создания километров шаблонного (boilerplate) кода.
5. Формат ответа: Выдавай ТОЛЬКО готовый к копированию код. Не пиши никаких вводных фраз и минимизируй текстовые пояснения после кода. Если пояснения нужны, пиши их супер-кратко и тезисно.
